# Floris TV Live Updates

## Full Project Context

## 1. Project Summary

This project is a **real-time warehouse operations dashboard and admin management system** built for a fulfillment/logistics workflow.

The goal of the system is to help a warehouse team:

- see which employees are assigned to which department
- manage subtasks inside each department
- track live task time and employee activity
- preserve assignment and task history for reporting
- export operational data into CSV and Excel dashboards

It is designed for a **TV/dashboard-style live operations screen** plus a **password-protected admin back office** where managers can control assignments, subtasks, employees, reporting, and historical data.

## 2. What the Product Solves

This system solves a common warehouse coordination problem:

- team leads need a simple live screen showing who is working where
- admins need to reassign staff quickly
- tasks must be created, updated, completed, archived, or deleted cleanly
- employee task history must remain available for review and reporting
- timing must be tracked in a business-relevant way rather than just raw browser time

In practice, this project acts like a **lightweight warehouse operations control center**.

## 5. Core Business Concepts

The app is built around these main concepts:

### Departments

Seeded warehouse departments include:

- Pick & Pack
- Replenishment
- Inbound
- Containers
- Returns
- Overige

Each department has:

- name
- color
- order

### Employees

Each employee can have:

- active/inactive state
- department assignment
- employee note / short description
- selected on-screen timer preference
- `Overige` lane selection (`internal` or `customers`)

### Assignments

Assignments connect employees to departments.

Key behaviors:

- active assignments are shown live on the dashboard
- reassignment creates historical trace
- assignment removals are preserved in history
- duplicate active assignments are deduped in UI and cleanup logic

### Subtasks

Subtasks represent real work inside departments.

Each subtask can include:

- title
- description
- assigned employees
- live/complete state
- attachment list
- final logged time override
- on-screen timer visibility
- history events

### Subtask Events

Subtask events preserve employee-task history, including:

- employee added
- employee removed
- task completed
- task reopened
- task deleted

This event stream is used for:

- employee task history
- reporting
- exports
- timeline reconstruction

## 8. Time Tracking Rules

The system uses **Netherlands local time** everywhere in the UI.

Time zone:

- `Europe/Amsterdam`

Important business rule:

- working time is counted from **09:00 NL**
- timers pause from **00:00 to 09:00 NL**

That means:

- timers do not continue counting overnight in raw continuous mode
- durations are aligned with the operational working day
- daylight saving is handled automatically through the Netherlands time zone

### Final Logged Time

Subtasks support a `Final logged time` override.

This is useful when:

- the auto timer is not the final reporting value
- a manager wants to manually set the official duration

If a final logged time exists:

- it overrides the live/automatic value in reporting and relevant UI

## 10. Data Persistence Model

The app uses **Firebase Firestore** as the primary database.

Main collections:

- `departments`
- `employees`
- `assignments`
- `subtasks`
- `subtaskEvents`
- `liveSync`

The Firestore layer is centralized in:

- `src/lib/firestore-db.js`

This abstraction handles:

- CRUD logic
- normalization
- event staging
- employee history assembly
- soft-delete behavior
- permanent delete behavior

## 11. Delete vs Archive Behavior

The app distinguishes between:

### Soft Delete / Remove From Screen

For normal task removal:

- the task disappears from active display
- it remains in Firestore
- history remains available

### Permanent Delete

Admin can also permanently delete a subtask:

- removes subtask document
- removes its event history
- clears any related timer references

This is useful for data cleanup when a task should truly be erased.

## 12. Real-Time / Multi-Screen Sync

The system includes a live-sync mechanism so that changes made in one place propagate to other open screens.

It uses:

- a Firestore-backed live sync token
- browser-side refresh hooks
- broadcast signaling for same-browser updates

This is implemented through:

- `src/app/api/live-sync/route.js`
- `src/lib/use-live-sync-refresh.js`
- `src/lib/live-sync-broadcast.js`

This means:

- dashboard screens
- admin screens
- employee detail screens

can react to data changes without requiring a manual refresh.
